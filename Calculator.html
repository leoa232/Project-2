Separation of Concerns

Separation of concerns is a software architecture design pattern/principle for divding an application into various parts, so each section acknowledges a separate concern. At its core, Separation of concerns is concerned about order. The ultimate purpose of separation of concerns is to create a structured system where each part peforms its role while maximizing the ability to adapt to change.

Separation of concerns is achieved by creating boundaries. A boundary is any logical or physical constraint which delineates a given set of responsibilities. Some examples of boundaries would include the use of methods, objects, components, and services to define core behavior within an application; projects, solutions, and folder hierarchies for s source organization; application layers and tiers for processing organization. Within the calculator program there are many examples of Seperation of Concerns being implemented. For example for source organization, several folder hiearchies were created. There was a folder specifically created for tests, another for calcuations (ie subtraction and addtion) and another folder that contained the main code for the calculator. We further divided these folders into subfolders, which would make it easier to pinpoint where something went wrong in a piece of code.

Some of the advantages of separation of Concerns implemented in software architecture incldue:

  • Entire program is easier to modify and maintain because it has been broken into purposepul parts.
  • The system becomes more stable as a result of the ease of maintenence.
  • The strategies required to ensure that each component only concerns itself with a single set of cohesive responsibilities often result in natural extensibility points.
  • The decoupling which results from requiring components to focus on a single purpose leads to components which are more easily reused in other systems, or different contexts within the same system.
